Methodological framework and protocol for mate-choice copying

Author

Eduardo S.A. Santos & Collaborators

Overview

This page documents all data decisions, processing steps, and planned analyses for the systematic review and meta-analysis of mate-choice copying (MCC) update across animal taxa.

Mate-choice copying occurs when an individual’s preference for a potential mate is influenced by observing the mate choices of others — a form of socially acquired preference that has been documented in fish, birds, mammals, and invertebrates.

This meta-analysis aims to:

  1. Estimate the overall effect size for MCC across species.
  2. Quantify between-study heterogeneity.
  3. Test for moderating variables (sex, taxon, experimental design).
NoteProject status

As of this version, the literature screening phase is concluded. This page has been updated with each phase of the pipeline.

Step 2 — Title-based deduplication

Titles are normalised to a canonical lowercase form before matching. When duplicates are found, the highest-quality record is retained.

Quality score (0–3): one point each for having a non-empty abstract, a non-empty author list, and a non-empty DOI.

Show R code 
# --- Normalise title ----------------------------------------------------------
articles[, title_lwr := str_to_lower(title)]
articles[, title_lwr := gsub("[[:punct:]]", " ", title_lwr)] # strip punctuation
articles[, title_lwr := gsub(" {2,}", " ", title_lwr)] # collapse whitespace
articles[, title_lwr := trimws(title_lwr)]

# --- Quality score (0–3) ------------------------------------------------------
articles[, has_abstract := ifelse(abstract == "" | is.na(abstract), 0L, 1L)]
articles[, has_authors := ifelse(authors == "" | is.na(authors), 0L, 1L)]
articles[, has_doi := ifelse(doi == "" | is.na(doi), 0L, 1L)]
articles[, quality := has_abstract + has_authors + has_doi]

# --- Concatenate source files per title group ---------------------------------
sources <- articles[,
    .(source_file = paste(sort(unique(source_file)), collapse = "; ")),
    by = .(title_lwr)
]

# --- Keep highest-quality record per title ------------------------------------
articles.title.dedupe <- articles[
    , .SD[which.max(quality), !c("source_file"), with = FALSE],
    by = .(title_lwr)
]

# --- Merge concatenated sources back in ---------------------------------------
articles.title.dedupe.mrg <- merge(
    articles.title.dedupe, sources,
    by = "title_lwr", all.x = TRUE
)

Step 3 — DOI-based deduplication

A second pass on the DOI field catches articles in which titles differ slightly between database exports but resolve to the same paper. A custom helper function split_conc() uniquely concatenates source strings that were themselves already concatenated in the title-dedup step.

Show R code 
# Helper: split already-concatenated source strings, deduplicate, re-join
split_conc <- function(x) {
    str_split(x, pattern = "; ") |>
        unlist() |>
        unique() |>
        sort() |>
        paste(collapse = "; ")
}

# Recalculate quality after title dedup (fields may have changed)
articles.title.dedupe.mrg[, has_abstract := ifelse(abstract == "" | is.na(abstract), 0L, 1L)]
articles.title.dedupe.mrg[, has_authors := ifelse(authors == "" | is.na(authors), 0L, 1L)]
articles.title.dedupe.mrg[, has_doi := ifelse(doi == "" | is.na(doi), 0L, 1L)]
articles.title.dedupe.mrg[, quality := has_abstract + has_authors + has_doi]

# Concatenate sources per DOI
sources <- articles.title.dedupe.mrg[,
    .(source_file = split_conc(source_file)),
    by = .(doi)
]

# Keep highest-quality record per DOI
articles.doi.dedupe <- articles.title.dedupe.mrg[
    , .SD[which.max(quality), !c("source_file"), with = FALSE],
    by = .(doi)
]

# Merge sources back in
articles.final <- merge(
    articles.doi.dedupe, sources,
    by = "doi", all.x = TRUE
)

Step 6 — Random shuffle and screener assignment

Articles are randomly shuffled before assignment to prevent ordering artefacts (e.g., alphabetical or chronological clustering within screener batches). A dynamic function generates near-equal assignment vectors for any team size.

set.seed(NULL) # no fixed seed — shuffled output files are the reproducible record

# Dynamic assignment: ceiling(total / n) reps, trimmed to total rows
create_assignments <- function(names, total_rows) {
    reps_needed <- ceiling(total_rows / length(names))
    full_vector <- rep(names, each = reps_needed)
    return(full_vector[seq_len(total_rows)])
}

n_total <- nrow(articles.final)

# Primary: ~33 articles per screener (495 / 15)
articles.final[, primary_screener := create_assignments(primary_list, n_total)]

# Secondary: ~248 articles per validator (495 / 2)
articles.final[, secondary_screener := create_assignments(secondary_list, n_total)]

# Verify balance
articles.final[, .(n = .N), by = .(primary_screener)]
articles.final[, .(n = .N), by = .(secondary_screener)]

Inclusion criteria

Articles are included if they:

  1. Report an experimental test of mate-choice copying (observer watches a demonstrator make a mate choice, then has the opportunity to choose between the same mates).
  2. Provide sufficient quantitative data for effect-size extraction (means + variance, or counts, or test statistics with sample sizes).
  3. Were published 2019 or later (pre-registration scope).
  4. Are written in English.

Inter-rater reliability

Disagreements between primary and secondary screeners will be resolved by discussion or a designated tie-breaker.

Data Extraction Pipeline

Included articles are sent to Google NotebookLM for AI-assisted data extraction via a four-step structured prompting protocol. All prompts are grounded by a MetaAnalysis_Reference_Sheet.pdf data dictionary containing few-shot examples of correct extractions.

WarningCache-bust rule

Each step must be run in a completely new NotebookLM chat window. Re-using a previous window causes the model to recycle earlier tables rather than reading the PDFs afresh.

Step 1 — Species & study identification

Goal: Establish taxonomic accuracy across all included papers.

Each PDF yields one row in a Markdown table with two columns:

Column Format Example
Study_ID FirstAuthorLastName_Year_CamelCaseJournal Belkina_2021_AnimalBehaviour
Species Full binomial nomenclature Poecilia reticulata

Common names alone are not accepted. The Study ID format is consistent across all four steps to serve as the primary join key.

Step 2 — Co-authorship & lab identification

Goal: Build a bipartite co-authorship network to identify “hub labs.”

Output is in long format — one row per author per paper — so that network edges can be constructed directly.

Column Description
Study_ID Join key
Author_Name Full name
Author_Position First / Last / Corresponding / Middle
Institutional_Affiliation University or institute
Lab_Identifier PILastName_Institution_City (e.g., Dukas_UniversityOfToronto_Toronto)
NoteLab identifier design

The last author is designated as the lab PI, and the same Lab_Identifier string is applied to every co-author row on the same paper. This links all authors to one “primary lab” node in the bipartite network, enabling correct degree-centrality calculation for hub-lab detection.

Step 3 — Trial-level data mapping

Goal: Map the internal experimental structure of each paper — how many distinct trials exist, where their data live, and whether subjects are independent.

Column Description
Study_ID Join key
Trial_ID Experiment name/number (e.g., “Experiment 1”, “Treatment A”)
Trial_Description 1–2 sentence summary of manipulation
Sample_Size_N Focal subjects in this trial
Subject_Independence “Independent Cohort” or “Repeated Measures”
Control_Group_Shared Yes/No + which trial (if Yes)
Data_Location Exact page and figure/table reference (e.g., “Table 1, Page 5”)

Step 4 — Quantitative extraction (multi-agent pipeline)

Goal: Extract the actual numbers and cross-check them within a single structured prompt.

Three-role prompting protocol

The extraction prompt instructs the language model to perform three sequential roles on each paper, so that every number is pulled, checked, and reconciled before it is written out. These are roles played by one model, not three separate programs:

  • Role 1 — Extractor: pulls the raw statistics (means, variances, counts, and test statistics) directly from the Methods and Results text and tables.
  • Role 2 — Verifier: cross-checks the extracted numbers, verifies sample independence, flags any standard-deviation vs. standard-error confusion, and re-reads the text to confirm no secondary trials were missed.
  • Role 3 — Arbiter: resolves any discrepancies between the Extractor and Verifier, standardises the reporting, and outputs the final data row.

The Verifier role checks explicitly for three systematic extraction errors:

  1. SD/SE confusion — a common error in manual extraction that biases heterogeneity estimates if uncorrected.
  2. Sample independence — flags whether the N used is independent across trials or from repeated measures on the same subjects.
  3. Missing trials — re-reads the Methods to confirm no secondary experiments were missed by Agent 1.

Effect Size Calculation

Following data extraction, effect sizes and their sampling variances are calculated using the reproducible Python pipeline located at scripts/1_effect_size_calculation_pipeline/calculate_effect_sizes.py.

The raw data is stored in the Excel spreadsheet data/1_effect_size_calculation_pipeline/Data Extraction Mate Choice Meta Analysis.xlsx inside the EduardoExtraction sheet. To support tracking, each row contains an identifierEffectSizeID column containing a unique sequential identifier for each control–experimental comparison (e.g., es0001, es0002). The Python pipeline dynamically resolves column locations by scanning the header row, computes the following effect size metrics, and saves them back to the spreadsheet:

1. Hedges’ \(g\) (Bias-Corrected Standardised Mean Difference)

Calculated depending on the data type extracted:

  • From Means & SDs: Using control and treatment means, standard deviations (or standard errors converted using \(SD = SE \times \sqrt{n}\)), and sample sizes.
  • From \(t\)-statistics: Supported for both paired/one-sample designs (Case 2, in the Python script) and independent-sample designs (Case 3, assuming an equal sample split if individual group sizes are unknown).

2. Odds Ratio (OR)

Calculated for categorical outcomes from \(2 \times 2\) contingency tables (\(A\), \(B\), \(C\), \(D\)), where the sampling variance of the log odds ratio is approximated using the delta method.

3. Odds Ratio \(\leftrightarrow\) Hedges’ \(g\) Cross-Conversion

For rows containing only one of the two metrics, the missing metric is derived using the logistic-normal approximation (Chinn 2000; Borenstein et al. 2011):

  • \(\text{OR} \rightarrow \text{Hedges' } g\): \(d = \ln(\text{OR}) \times \frac{\sqrt{3}}{\pi}\) and then bias-corrected to \(g\) using the \(J\) correction factor.
  • \(\text{Hedges' } g \rightarrow \text{OR}\): \(\ln(\text{OR}) = g \times \frac{\pi}{\sqrt{3}}\) and \(\text{Var}(\ln\text{OR}) = \text{Var}(g) \times \frac{\pi^2}{3}\).

To execute the calculations and update the workbook, run:

python scripts/1_effect_size_calculation_pipeline/calculate_effect_sizes.py

Planned Analyses

All models will be implemented in R. Our pre-registration details are available in Santos et al. (2026).

Meta-analytic models

Multi-level random-effects meta-analysis to estimate the overall effect size for MCC, accounting for trials nested within papers nested within labs.

Package: metafor.

Heterogeneity will be quantified with \(I^2\) and \(\tau^2\) at each level.

Phylogenetic comparative methods

Species non-independence is controlled via a phylogenetic covariance matrix incorporated into the meta-analytic model (metafor + ape). This accounts for the possibility that closely related species share similar MCC propensity due to shared evolutionary history.

Publication bias

Funnel plot asymmetry, and Egger’s regression test to assess publication bias. Time-lag bias will be examined by plotting effect size against publication year.

Moderator analyses

Meta-regression models will test whether effect sizes vary by:

  • Sex of the copying individual (female vs. male)
  • Taxonomic group (fish, bird, mammal, invertebrate)
  • Experimental design (before-and-after vs. no pre-test)
  • Copying mechanism (individual vs. generalized)

Sensitivity analyses

  • Inclusion of pre-2019 records.
  • Leave-one-out analysis for influential studies.

```{r}
# Helper: split already-concatenated source strings, deduplicate, re-join
split_conc <- function(x) {
    str_split(x, pattern = "; ") |>
        unlist() |>
        unique() |>
        sort() |>
        paste(collapse = "; ")
}

# Recalculate quality after title dedup (fields may have changed)
articles.title.dedupe.mrg[, has_abstract := ifelse(abstract == "" | is.na(abstract), 0L, 1L)]
articles.title.dedupe.mrg[, has_authors := ifelse(authors == "" | is.na(authors), 0L, 1L)]
articles.title.dedupe.mrg[, has_doi := ifelse(doi == "" | is.na(doi), 0L, 1L)]
articles.title.dedupe.mrg[, quality := has_abstract + has_authors + has_doi]

# Concatenate sources per DOI
sources <- articles.title.dedupe.mrg[,
    .(source_file = split_conc(source_file)),
    by = .(doi)
]

# Keep highest-quality record per DOI
articles.doi.dedupe <- articles.title.dedupe.mrg[
    , .SD[which.max(quality), !c("source_file"), with = FALSE],
    by = .(doi)
]

# Merge sources back in
articles.final <- merge(
    articles.doi.dedupe, sources,
    by = "doi", all.x = TRUE
)
```

## Data Extraction Pipeline {#sec-extraction}

Included articles are sent to **Google NotebookLM** for AI-assisted data extraction
via a four-step structured prompting protocol. All prompts are grounded by a
`MetaAnalysis_Reference_Sheet.pdf` data dictionary containing few-shot examples of
correct extractions.

::: {.callout-warning}
## Cache-bust rule
Each step must be run in a **completely new** NotebookLM chat window. Re-using a
previous window causes the model to recycle earlier tables rather than reading the
PDFs afresh.
:::

### Step 1 — Species & study identification

**Goal:** Establish taxonomic accuracy across all included papers.

Each PDF yields one row in a Markdown table with two columns:

| Column | Format | Example |
|---|---|---|
| `Study_ID` | `FirstAuthorLastName_Year_CamelCaseJournal` | `Belkina_2021_AnimalBehaviour` |
| `Species` | Full binomial nomenclature | *Poecilia reticulata* |

Common names alone are not accepted. The Study ID format is consistent across all
four steps to serve as the primary join key.

### Step 2 — Co-authorship & lab identification

**Goal:** Build a bipartite co-authorship network to identify "hub labs."

Output is in **long format** — one row per author per paper — so that network edges
can be constructed directly.

| Column | Description |
|---|---|
| `Study_ID` | Join key |
| `Author_Name` | Full name |
| `Author_Position` | First / Last / Corresponding / Middle |
| `Institutional_Affiliation` | University or institute |
| `Lab_Identifier` | `PILastName_Institution_City` (e.g., `Dukas_UniversityOfToronto_Toronto`) |

::: {.callout-note}
## Lab identifier design
The **last author** is designated as the lab PI, and the same `Lab_Identifier` string
is applied to every co-author row on the same paper. This links all authors to one
"primary lab" node in the bipartite network, enabling correct degree-centrality
calculation for hub-lab detection.
:::

### Step 3 — Trial-level data mapping

**Goal:** Map the internal experimental structure of each paper — how many distinct
trials exist, where their data live, and whether subjects are independent.

| Column | Description |
|---|---|
| `Study_ID` | Join key |
| `Trial_ID` | Experiment name/number (e.g., "Experiment 1", "Treatment A") |
| `Trial_Description` | 1–2 sentence summary of manipulation |
| `Sample_Size_N` | Focal subjects in this trial |
| `Subject_Independence` | "Independent Cohort" or "Repeated Measures" |
| `Control_Group_Shared` | Yes/No + which trial (if Yes) |
| `Data_Location` | Exact page and figure/table reference (e.g., "Table 1, Page 5") |

### Step 4 — Quantitative extraction (multi-agent pipeline)

**Goal:** Extract the actual numbers and cross-check them within a single structured
prompt.

#### Three-role prompting protocol

The extraction prompt instructs the language model to perform three sequential roles on
each paper, so that every number is pulled, checked, and reconciled before it is written
out. These are roles played by one model, not three separate programs:

*   **Role 1 — Extractor:** pulls the raw statistics (means, variances, counts, and test statistics) directly from the Methods and Results text and tables.
*   **Role 2 — Verifier:** cross-checks the extracted numbers, verifies sample independence, flags any standard-deviation vs. standard-error confusion, and re-reads the text to confirm no secondary trials were missed.
*   **Role 3 — Arbiter:** resolves any discrepancies between the Extractor and Verifier, standardises the reporting, and outputs the final data row.

**The Verifier role checks explicitly for three systematic extraction errors:**

1. **SD/SE confusion** — a common error in manual extraction that biases heterogeneity
   estimates if uncorrected.
2. **Sample independence** — flags whether the N used is independent across trials or
   from repeated measures on the same subjects.
3. **Missing trials** — re-reads the Methods to confirm no secondary experiments
   were missed by Agent 1.

#### Final output columns

| Column | Content |
|---|---|
| `Study_ID` | Author_Year_JournalName |
| `Trial_ID` | Specific experiment or condition |
| `Response_Metric` | What was measured |
| `Validated_Exp_Group_Data` | Mean ± SD/SE, Counts, or Test Stat |
| `Validated_Control_Group_Data` | Mean ± SD/SE, Counts, or NA |
| `SD_or_SE` | Explicit variance metric label |
| `Agent1_Initial_Pull` | Brief summary of original extraction |
| `Agent2_Verification_Notes` | Corrections made, or "No missing trials found" |
| `Verification_Location` | Exact page and table/figure number |

## Effect Size Calculation {#sec-effect-sizes}

Following data extraction, effect sizes and their sampling variances are calculated using the reproducible Python pipeline located at [scripts/1_effect_size_calculation_pipeline/calculate_effect_sizes.py](file:///Users/eduardosantos/Documents/Repos/mate_choice_meta/scripts/1_effect_size_calculation_pipeline/calculate_effect_sizes.py). 

The raw data is stored in the Excel spreadsheet [data/1_effect_size_calculation_pipeline/Data Extraction Mate Choice Meta Analysis.xlsx](file:///Users/eduardosantos/Documents/Repos/mate_choice_meta/data/1_effect_size_calculation_pipeline/Data%20Extraction%20Mate%20Choice%20Meta%20Analysis.xlsx) inside the `EduardoExtraction` sheet. To support tracking, each row contains an `identifierEffectSizeID` column containing a unique sequential identifier for each control–experimental comparison (e.g., `es0001`, `es0002`). The Python pipeline dynamically resolves column locations by scanning the header row, computes the following effect size metrics, and saves them back to the spreadsheet:

### 1. Hedges' $g$ (Bias-Corrected Standardised Mean Difference)
Calculated depending on the data type extracted:

* **From Means & SDs:** Using control and treatment means, standard deviations (or standard errors converted using $SD = SE \times \sqrt{n}$), and sample sizes.
* **From $t$-statistics:** Supported for both paired/one-sample designs (Case 2, in the Python script) and independent-sample designs (Case 3, assuming an equal sample split if individual group sizes are unknown).

### 2. Odds Ratio (OR)
Calculated for categorical outcomes from $2 \times 2$ contingency tables ($A$, $B$, $C$, $D$), where the sampling variance of the log odds ratio is approximated using the delta method.

### 3. Odds Ratio $\leftrightarrow$ Hedges' $g$ Cross-Conversion
For rows containing only one of the two metrics, the missing metric is derived using the logistic-normal approximation [@chinnSimpleMethodConverting2000; @borensteinIntroductionMetaanalysis2011]:

* **$\text{OR} \rightarrow \text{Hedges' } g$**: $d = \ln(\text{OR}) \times \frac{\sqrt{3}}{\pi}$ and then bias-corrected to $g$ using the $J$ correction factor.
* **$\text{Hedges' } g \rightarrow \text{OR}$**: $\ln(\text{OR}) = g \times \frac{\pi}{\sqrt{3}}$ and $\text{Var}(\ln\text{OR}) = \text{Var}(g) \times \frac{\pi^2}{3}$.

To execute the calculations and update the workbook, run:
```bash
python scripts/1_effect_size_calculation_pipeline/calculate_effect_sizes.py
```

## Planned Analyses {#sec-analyses}

All models will be implemented in R. Our pre-registration details are available in Santos et al. [-@santosMatechoiceCopyingNonhuman2026].

### Meta-analytic models

Multi-level random-effects meta-analysis to estimate the overall effect size for MCC, accounting for trials nested within papers nested within labs.

Package: `metafor`.

**Heterogeneity** will be quantified with $I^2$ and $\tau^2$ at each level.

### Phylogenetic comparative methods

Species non-independence is controlled via a phylogenetic covariance matrix
incorporated into the meta-analytic model (`metafor` + `ape`). This accounts for the possibility that closely related species share similar MCC propensity due to shared evolutionary history.

### Publication bias

Funnel plot asymmetry, and Egger's regression test to assess publication bias.
Time-lag bias will be examined by plotting effect size against publication year.

### Moderator analyses

Meta-regression models will test whether effect sizes vary by:

- Sex of the copying individual (female vs. male)
- Taxonomic group (fish, bird, mammal, invertebrate)
- Experimental design (before-and-after vs. no pre-test)
- Copying mechanism (individual vs. generalized)

### Sensitivity analyses

- Inclusion of pre-2019 records.
- Leave-one-out analysis for influential studies.
